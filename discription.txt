EXPANCE TRACKER PROJECT DESCRIPTION
===================================

1. Project Overview
-------------------
Expance Tracker is a full-stack personal finance management web application. It helps a user manage daily income, expenses, wallets, categories, budgets, reminders, dashboard summaries, and financial analytics from one authenticated web interface.

The project is built as a server-rendered Node.js application using Express.js, EJS templates, MongoDB, and a Bootstrap-based admin UI. It is designed for single-user-session flows where each authenticated user can manage their own financial records separately.

The application focuses on these main goals:
- Track income and expense transactions.
- Organize transactions by wallet and category.
- Manage multiple wallets such as bank, cash, and digital wallets.
- Create category-based and overall budgets.
- Monitor spending against budgets.
- Show dashboard summaries and analytics.
- Manage reminders for EMI, bills, borrowed money, lent money, and tasks.
- Support OTP-based email login.
- Store user preferences and notification settings.

2A. Unique Technologies, Services, and Special Features
------------------------------------------------------
This project is not only a basic income-expense CRUD application. It includes several unique and useful technical/service-level features that make it more complete as a personal finance system.

Passwordless OTP Login:
- The project uses email OTP authentication instead of a normal password login.
- OTPs are generated, stored, verified, expired, and blocked after too many failed attempts.
- This makes login simpler for users because they do not need to remember a password.
- Nodemailer is used as the email delivery service for sending OTP codes.

MongoDB-backed Session Storage:
- User sessions are not stored only in server memory.
- connect-mongo stores sessions in MongoDB.
- This is useful for persistence and is better than default in-memory sessions for a real application.

Automatic Expense Creation from Reminders:
- EMI and bill reminders can be marked as paid.
- When marked paid, the application automatically creates an expense transaction.
- This connects the reminder module directly with the transaction module.
- It reduces duplicate user work because the user does not need to manually add the same bill as an expense again.

Recurring Reminder Scheduling:
- EMI/Bill reminders support repeat cycles such as monthly, quarterly, and yearly.
- When a repeating reminder is marked paid, the reminder is moved to the next due date instead of being permanently closed.
- This is useful for real recurring financial obligations.

Wallet Transfer-aware Balance System:
- Wallet balances are not calculated only from transactions.
- The app considers opening balance, income, expenses, incoming transfers, and outgoing transfers.
- This makes wallet balances more realistic when users move money between cash, bank, and digital wallets.

Category and Wallet Reference Synchronization:
- If a category is renamed, related transaction records can be updated with the new category name.
- If a wallet is renamed, related transaction and wallet transfer references can be updated.
- This avoids broken historical records after renaming important master data.

Budget Intelligence:
- The budget module supports overall budgets and category-level budgets.
- It calculates used percentage, remaining amount, over-budget amount, and alert status.
- It can use monthly budgets, custom date ranges, and fallback monthly budget preferences from settings.

Dashboard-level Financial Intelligence:
- The dashboard combines transactions, wallets, budgets, reminders, settings, and categories into one summary.
- It shows savings rate, current month income/expense, budget progress, reminder alerts, upcoming tasks, wallet balances, and spending breakdowns.
- This makes the dashboard more like a financial control center instead of a simple stats page.

Header Reminder Notifications:
- Reminder alerts are loaded globally and shown in the header.
- The notification count is based on due-today and overdue reminders.
- User notification preferences control whether bill reminders are enabled.

Bill Image Attachment Support:
- Transactions can include bill image data.
- The app validates file type and approximate file size.
- Supported image types include JPG, PNG, WEBP, and GIF.

Starter Category Seeding:
- The category module automatically creates useful starter income and expense categories for new users.
- This improves first-time user experience because the user can start adding transactions quickly.

Finance-focused Modular Architecture:
- The project uses separate route, controller, collection, utility, and view layers.
- This structure keeps finance logic organized and makes the project easier to extend.
- Important domain logic such as budgets, reminders, category visuals, wallet balances, and OTP authentication is separated into reusable helper modules.

4. Application Startup Flow
---------------------------
The application starts from server.js.

Startup steps:
1. Load environment variables using dotenv.
2. Create an Express app.
3. Set EJS as the view engine.
4. Enable URL-encoded form parsing with a 10MB limit.
5. Serve static files from the public folder.
6. Configure express-session.
7. Store sessions in MongoDB using connect-mongo.
8. Enable flash messages.
9. Load current user details and reminder notifications into res.locals.
10. Apply cache-control middleware.
11. Mount all route modules.
12. Connect to MongoDB.
13. Start the HTTP server on PORT or 3000.

Important note:
- config/db.js currently connects directly to mongodb://127.0.0.1:27017 and uses database expanceTracker.
- Session storage uses process.env.MONGO_URI through connect-mongo.
- For consistent deployment, both should point to the same MongoDB environment.


5. Authentication System
------------------------
The application uses email OTP login instead of password login.

Authentication features:
- User enters email on the login page.
- Email is normalized to lowercase.
- Email format is validated.
- OTP is generated using utils/generateOTP.js.
- OTP is saved in the otps collection.
- OTP email is sent using utils/emailService.js and nodemailer.
- The email is also saved/updated in emailLeads collection.
- Active OTP protection prevents repeated OTP spam.
- OTP verification requires a 6-digit numeric code.
- Invalid attempts are counted.
- OTP is blocked after too many failed attempts.
- Expired OTPs are rejected.
- On success, the user is found or created in users collection.
- OTP is marked as verified.
- Email lead is marked as verified.
- A fresh session is created.
- Session details are stored in session history.
- User is redirected to dashboard.

Logout:
- Logout updates session history.
- Express session is destroyed.
- Session cookie is cleared.
- User is redirected to login.

- Requires person name, amount, borrow date, and return date.

- Can be marked returned.

Lent money behavior:
- Requires person name, amount, given date, and expected return date.
- Expected return date cannot be before given date.
- Can be marked received.

Task behavior:
- Requires title and due date.
- Can be marked completed.

- Total active reminders

16. Security and Validation
---------------------------
Security-related behavior:
- Auth middleware protects private pages.
- alreadyAuth middleware prevents logged-in users from returning to login.
- Sessions are stored in MongoDB.
- Session cookie is httpOnly.
- sameSite is set to lax.
- secure cookie is enabled only in production.
- Flash messages avoid silent failures.
- OTP attempts are limited.
- Blocked OTPs prevent brute-force style repeated verification.
- User records are scoped by userId.
- Most update/delete operations check both item id and userId.

Validation examples:
- Email format validation.
- OTP must be 6 digits.
- Transaction amount must be greater than 0.
- Transaction type must be income or expense.
- Category must exist.
- Wallet must exist.
- Category type must match transaction type.
- Wallet transfer amount must be greater than 0.
- Transfer source and destination wallets must be different.
- Source wallet must have enough balance.
- Category budget limit must be 0 or greater.
- Reminder dates must be valid.
- Return dates cannot be before borrow/given dates.
- Bill image file type and size are validated.

19. Important Business Rules
----------------------------
- Every user sees only their own data.
- Income and expense categories are separate.
- A transaction category must match the transaction type.
- A transaction must belong to an existing wallet.
- Wallets in use cannot be deleted.
- Categories in use cannot be deleted.
- Wallet transfers require enough balance in the source wallet.
- Budget status is based on selected period spending.
- Category budget alerts appear when spending is near or over budget.
- EMI/Bill reminder payment creates an expense transaction automatically.
- Repeating paid reminders move to next due date instead of staying completed.
- Reminder notifications depend on user notification settings.

23. Summary
-----------
Expance Tracker is a personal finance management application built with Node.js, Express.js, EJS, MongoDB, and Bootstrap assets. It supports OTP login, dashboard reporting, transaction tracking, wallet management, category management, budgeting, analytics, reminders, notifications, and user settings. The project is organized in a maintainable MVC-like structure and uses MongoDB collection helper modules for data access. It is suitable as a complete personal finance tracker and can be extended further with exports, scheduled notifications, stronger security, and automated testing.
